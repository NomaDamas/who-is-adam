# ICML 2026 PDF 리뷰 스킬 제품 제안

## 현재 상태

이 문서는 `who-is-adam`의 제품 제안서입니다. 현재 체크포인트는 구현 전 문서화 단계이며, 아래 흐름과 인터페이스는 구현 예정 동작을 설명합니다. 실제 CLI, Python 패키지, 테스트 fixture, 외부 제공자 클라이언트는 후속 체크포인트에서 별도로 구현되어야 합니다.

## 배경

ICML 2026 Main Track 리뷰는 논문 내용 평가뿐 아니라 제출 형식, 익명성, LLM 사용 정책, 비밀유지, 행동 강령, 인용 근거 확인을 함께 요구합니다. 공식 Main Track 리뷰 양식은 Soundness, Presentation, Significance, Originality를 1-4로, Overall Recommendation을 1-6으로, Confidence를 1-5로 기록합니다. 논문 저자의 프롬프트 인젝션은 금지되어 있고, 리뷰어는 배정된 LLM 정책을 따라야 합니다.

## 사용자 문제

리뷰어는 제한된 시간 안에 다음 작업을 반복해야 합니다.

- PDF가 ICML 2026 Main Track 제한에 맞는지 확인한다.
- 본문 주장과 참고문헌이 실제 근거와 맞는지 점검한다.
- 선행 연구와의 차이를 판단하되, 공개 근거가 없는 과거 리뷰 내용을 만들지 않는다.
- 리뷰 초안이 공식 필드 순서와 점수 범위를 벗어나지 않게 한다.
- 논문 내부의 악성 지시문이 LLM 또는 자동화 도구의 행동을 바꾸지 못하게 한다.

## 제안하는 해결책

`who-is-adam`은 로컬 PDF 하나를 입력으로 받아, 안전 게이트와 근거 검증을 통과한 경우에만 ICML Main Track 형식의 Markdown 리뷰 초안을 저장하는 Python CLI/라이브러리로 계획됩니다. 도구는 리뷰어에게 구조화된 근거, 위험 신호, 인용 검증 상태, 독립 전문 관점의 평가, 합성 리뷰를 제공하지만 최종 판단과 제출은 사람이 수행합니다.

## 처리 흐름

예정 처리 흐름은 다음 순서를 따릅니다.

1. **PDF 입력**: 운영자가 로컬 단일 PDF와 출력 디렉터리, 배정된 LLM 정책, 행동 강령 확인을 전달합니다.
2. **구조 추출**: 제목, 초록, 섹션, 페이지, 표, 그림, 수식, 참고문헌, 텍스트 span과 추출 품질 지표를 수집합니다.
3. **품질/인젝션 게이트**: 낮은 텍스트 밀도, OCR 실패, 손상/암호화 PDF, 프롬프트 인젝션 의심 패턴을 확인하고 필요하면 거절합니다.
4. **ICML 규정 검사**: 단일 PDF, 50MB 제한, 메인 본문 8쪽 제한, 익명성 신호, LaTeX 형식 관련 휴리스틱을 점검합니다.
5. **인용 검증**: Crossref, Semantic Scholar, arXiv로 참고문헌의 제목/연도/DOI/arXiv ID/venue 일치 상태를 확인합니다.
6. **선행연구/OpenReview 근거**: 직접 비교, 개선, 우월성 주장과 연결된 핵심 선행연구를 최대 다섯 개까지 선택하고, 공개 OpenReview 근거가 있을 때만 과거 강점/약점 비교를 사용합니다.
7. **독립 전문 리뷰**: 여러 전문 관점이 서로의 초안을 보지 않고 PDF 근거와 검증 결과만 사용해 평가합니다.
8. **종합**: 합의점, 충돌, 소수 의견, 불확실성을 보존해 공식 ICML 필드와 점수 범위로 합성합니다.
9. **Markdown 저장**: 정규화된 논문 제목 디렉터리에 `<normalized_title>_review_{n}.md` 형식으로 저장합니다.

## 리뷰 품질 원칙

- **Evidence before judgment**: 모든 주요 평가는 논문 span, 페이지, 섹션 또는 외부 출처 상태와 연결합니다.
- **Fail closed**: 안전하지 않거나 읽을 수 없는 입력은 추측 리뷰를 만들지 않고 거절합니다.
- **Official schema and scales only**: ICML Main Track 공식 필드와 점수 범위만 허용합니다.
- **Independent specialists**: 전문 리뷰 단계는 peer output을 공유하지 않아 군집 사고와 상호 오염을 줄입니다.
- **Deterministic verification**: 테스트는 fake LLM, 고정 timestamp/seed, ReportLab PDF fixture, `respx` HTTP mock을 사용해 네트워크 없이 재현 가능해야 합니다.
- **No fabricated evidence**: 외부 API가 실패하거나 OpenReview 근거가 없으면 `unavailable`로 남깁니다.

## 사람이 최종 판단한다

이 도구는 리뷰어를 보조하는 분석/초안 생성 도구입니다. ICML의 공식 결정, 점수 확정, 리뷰 제출, 이해상충 판단, 윤리적 판단은 사람이 수행해야 합니다. 도구가 저장한 Markdown은 검토 가능한 초안이며, 자동 제출물이나 공식 판정이 아닙니다.

## ICML Main Track 한계

- Position Track은 범위 밖입니다.
- 제출 PDF는 단일 파일, 최대 50MB, 메인 본문 최대 8쪽이어야 합니다.
- 참고문헌과 부록은 본문 뒤에 허용되지만, 본문/부록 경계가 불명확하면 통과로 꾸미지 않고 `unknown` 또는 경고로 남깁니다.
- 익명성/형식/페이지 위반은 자동 거절 가능성이 있는 신호로 표시합니다.
- 리뷰 출력은 Soundness/Presentation/Significance/Originality 1-4, Overall Recommendation 1-6, Confidence 1-5의 공식 범위만 사용합니다.

## 비목표

- ICML 또는 OpenReview에 리뷰를 제출하지 않습니다.
- 저자를 대신해 논문을 수정하지 않습니다.
- 공개 근거가 없는 OpenReview 과거 리뷰 내용을 생성하지 않습니다.
- 모든 스캔 PDF를 OCR로 복구한다고 보장하지 않습니다.
- 테스트에서 실제 네트워크, 실제 API 키, 현재 날짜, 로컬 Tesseract 설치에 의존하지 않습니다.

## 성공 기준 요약

성공한 구현은 다음 조건을 만족해야 합니다.

- 한국어 문서가 제품 목적, 워크플로, 거절 의미, 증거 정책, 운영 설정, 출력 이름, 단계별 구현 계획을 먼저 설명한다.
- 안전/품질 게이트가 리뷰 생성보다 먼저 실행된다.
- 인용 및 OpenReview 근거 부재를 추측하지 않고 명시적으로 `unavailable` 처리한다.
- 공식 ICML Main Track 필드와 점수 범위를 코드와 테스트에서 검증한다.
- 오프라인 테스트 모드가 네트워크 호출 없이 결정적 결과를 만든다.
